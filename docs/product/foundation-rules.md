## 1. 统一身份与会话

### 1.1 身份主体

1. 系统中一个自然人对应唯一 `**identity_subject`**（身份主体）。
2. 手机号是主体归并键：同一手机号仅对应一个有效主体。
3. 主体类型为 `person`（自然人）；系统级主体如有，类型为 `system`，不参与客户端/员工端登录。

### 1.2 双账号模型

同一 `identity_subject` 下可同时存在：


| 逻辑实体                   | 用途           | 登录端         |
| ---------------------- | ------------ | ----------- |
| `**staff_account`**    | 员工账号（客服、管理员） | 员工端 / 平台管理端 |
| `**customer_account**` | 客户账号         | 客户端         |


规则：

- 员工端账号与客户端账号**分入口登录**，底层共享统一认证体系。
- 账号禁用/注销后，对应端登录立即失效；历史业务数据保留策略按账号类型在 Backlog 中细化。

### 1.3 会话与认证


| 规则     | 说明                                                         |
| ------ | ---------------------------------------------------------- |
| 认证方式   | JWT + Refresh Token 轮换                                     |
| 多会话    | 支持单账号多会话；可配置最大会话数与会话 TTL                                   |
| 权限变更失效 | 用户角色、业务域成员关系、账号状态变更后，**已签发 Access Token 须失效**，用户重新登录后获取新权限 |
| 登录审计   | 成功/失败登录均写入 `**login_log`**，含失败原因、终端、`client_code` 等，便于安全审计 |


### 1.4 客户端 / 员工端切换（规划规则）

若同一主体已在当前环境登录员工端，打开客户端时，系统应识别有效主体会话并支持**免密进入客户端**（具体交互与 Story 在 Backlog 中排期）。

### 1.5 客户入域

1. 客户必须对某 `**business_domain`** 具备有效访问关系后，方可使用该业务域的工单、咨询等服务。
2. 入域途径受 **业务域注册配置、业务域邀请配置** 及 **管理员添加** 规则约束（见第 2.2、2.3 节）。
3. 未入域客户访问该业务域业务能力时，系统须拒绝并给出**中文**提示。

### 1.6 逻辑名与实现别名


| 逻辑名                | 说明   | 实现别名（过渡期）                                     |
| ------------------ | ---- | --------------------------------------------- |
| `identity_subject` | 身份主体 | —                                             |
| `staff_account`    | 员工账号 | 部分路径仍称 `user_account`，**新 Story 统一 staff 语义** |
| `customer_account` | 客户账号 | —                                             |

## 2. 多业务域隔离

### 2.1 隔离单元

`**business_domain`（业务域）** 是资源隔离的基本单元。每个业务域拥有独立的工单、咨询、成员、配置与权限范围。

产品支持**多个业务域**并存；MVP 北极星以 **至少 1 个业务域** 完成工单闭环为准（见 `[vision.md](./vision.md)`）。

### 2.2 域策略

#### 2.2.1 可见策略（不变）


| 可选值                        | 作用              |
| -------------------------- | --------------- |
| `公开` / `仅域内客户可见` / `仅渠道可见` | 控制谁可以**发现**该业务域 |


#### 2.2.2 入域三配置（独立维度）

以下三项**相互独立**，域管理员分别配置；不再使用单一的「注册策略」三选一表述（见 2.2.4 过渡映射）。


| 配置项         | 可选值          | 作用                                      |
| ----------- | ------------ | --------------------------------------- |
| **业务域注册配置** | 允许注册 / 不允许注册 | 客户是否可在客户端**自助注册并加入**该业务域                |
| **业务域邀请配置** | 允许邀请 / 不允许邀请 | 是否可使用**邀请码**加入该业务域                      |
| **业务域添加**   | （无域级开关）      | 具备**业务域客户添加**权限的管理人员可直接添加客户，**无需额外域配置** |


**逻辑字段名（目标态）**：`registration_enabled`、`invitation_enabled`，取值 `allowed` / `disallowed`。

#### 2.2.3 入域行为规则


| 规则 ID     | 规则                  | 客户端 / API 行为                                                     |
| --------- | ------------------- | ---------------------------------------------------------------- |
| **DR-01** | 业务域注册配置 = **不允许注册** | **隐藏**该业务域的注册入口/注册按钮；若因异常仍展示入口，提交时须拒绝并返回**中文**提示（如「该业务域不允许自助注册」） |
| **DR-02** | 业务域邀请配置 = **不允许邀请** | 邀请码入域拒绝 + **中文**提示（如「该业务域不支持邀请码入域」）                              |
| **DR-03** | 注册与邀请均不允许           | 客户仍可通过 **管理员添加** 入域（DR-04）                                       |
| **DR-04** | **业务域添加**           | 不受注册/邀请配置限制；仅校验操作者是否具备业务域客户添加相关**权限项**                           |
| **DR-05** | 注册与邀请               | **可独立组合**（四种组合均合法）                                               |


#### 2.2.4 历史迁移说明：`registration_policy` → 双字段

> **状态**：已于 Flyway **`V202605250001`** 落地；库表已使用 `registration_enabled` / `invitation_enabled`，旧列 `registration_policy` 已 **DROP**。  
> 下列映射仅供**自旧库升级**或阅读历史脚本时对照。

| 旧值 `registration_policy` | 业务域注册配置 | 业务域邀请配置 |
| ------------------------ | ------- | ------- |
| `open`                   | 允许注册    | 允许邀请    |
| `invitation_only`        | 不允许注册   | 允许邀请    |
| `admin_only`             | 不允许注册   | 不允许邀请   |

登记见 [`architecture/database-increment-plan.md`](../architecture/database-increment-plan.md)。

### 2.3 数据隔离（MVP）

1. **方案**：共享表 + `**business_domain_id`** 逻辑隔离（单库多域）。
2. **强制约束**：凡读取或写入带业务域属性的数据，必须在数据访问层带 `**business_domain_id`** 过滤或赋值。
3. **跨域禁止**：A 业务域身份不得访问 B 业务域工单、咨询、域配置、成员列表等；越权时 API 返回 **403** 或业务空集。

### 2.4 域生命周期


| 状态  | 行为摘要                                    |
| --- | --------------------------------------- |
| 启用  | 业务域内服务可正常使用                             |
| 禁用  | 客户端不可新建该业务域服务请求；已有数据只读/处理规则在 Backlog 细化 |


域创建、策略修改、禁用/删除等管理动作须可追溯（审计日志）。

## 3. 权限体系

### 3.1 双层权限模型


| 层级       | 典型角色                                             | 范围                   |
| -------- | ------------------------------------------------ | -------------------- |
| **平台级**  | `platform_admin`                                 | 创建业务域、全局员工、系统安全与平台菜单 |
| **业务域级** | `super_admin` / `domain_admin` / `agent` + 自定义角色 | 入域单业务域内配置、成员、工单与咨询处理 |


### 3.2 域内预设角色


| 角色代码           | 职责摘要         |
| -------------- | ------------ |
| `super_admin`  | 业务域内最高权限     |
| `domain_admin` | 业务域内常规配置与人员  |
| `agent`        | 基础客服：工单、咨询处理 |


域管理员可创建自定义业务域角色，**每业务域最多 20 个**（不含预设角色的自定义数量限制以 Backlog 为准）。

### 3.3 权限项

1. **格式**：`模块:操作`，例如 `ticket:delete`。
2. **数据范围**：后缀 `:self`、`:domain_all`。
3. **菜单与按钮**：未授权菜单不可见；未授权 API 返回 **403 + 中文**错误信息。
4. **业务域客户添加**：由权限项控制（如 `domain:customer:add` 或项目等价码）；**不**在业务域上单独设「添加配置」开关。

### 3.4 权限存储与生效


| 规则    | 说明                         |
| ----- | -------------------------- |
| 角色授权  | 用户 → 角色 → 权限项 / 菜单         |
| 生效时机  | 绑定或权限变更后，**当前会话 token 失效** |
| 个性化直授 | MVP 若未交付则后续；若实现须审计原因       |


### 3.5 全局安全

- 敏感词：全平台词库，命中拦截并告警（详见第 8 章 / 架构文档）。
- 审计：管理操作、权限变更写入 `**audit_log` / `operation_log`**，不可删除。
- 高危操作：删除业务域、全量导出等须二次密码验证。

## 6. 工单状态机

### 6.1 配置层级

- 各**工单类型**可配置独立状态流，须服从本章全局约束。
- 动态字段与 Formily 边界见 ADR / Backlog（M11），本章仅规定状态语义。

### 6.2 状态属性标签

域管理员配置每个状态时须指定：


| 属性         | 选项                  | 说明               |
| ---------- | ------------------- | ---------------- |
| **状态类型**   | `进行中` / `暂停` / `终态` | 决定 SLA 与能否流转     |
| **允许客户撤回** | 是 / 否               | 仅 **进行中** 可设为「是」 |
| **为「已解决」** | 是 / 否               | 用于满意度评价等后续流程     |


### 6.3 全局约束

1. 每个工单类型的状态流中 **至少有一个终态**，否则**不允许保存**（TR-01）。
2. **终态不可逆**：进入终态后不可再变更；问题复发须**新建工单**并可选关联历史工单号。
3. 系统根据目标状态的**状态类型**自动执行 SLA 动作（续表 / 暂停 / 停止）。

### 6.4 预置默认状态流（逻辑名）

```
待处理 → 处理中 ⇄ 待客户补充 → 处理中
              ├→ 已解决 → 已关闭
              ├→ 已取消（客户撤回）
              └→ 已合并（重复工单）
```


| 逻辑状态                  | 状态类型 | 允许客户撤回 |
| --------------------- | ---- | ------ |
| 待处理                   | 进行中  | 是      |
| 处理中                   | 进行中  | 否      |
| 待客户补充                 | 暂停   | 否      |
| 已解决 / 已关闭 / 已取消 / 已合并 | 终态   | 否      |


### 6.5 客户撤回

- 仅当工单处于 **进行中** 且当前状态配置 **允许客户撤回** 时可撤回（TR-03）。
- 撤回后进入终态「已取消」或项目约定的等价终态。

### 6.6 SLA 联动（概要）


| 状态类型 | SLA         |
| ---- | ----------- |
| 进行中  | 继续计时        |
| 暂停   | 暂停计时（TR-05） |
| 终态   | 永久停止        |


完整 SLA 规则见第 8 章（M07）。

### 6.7 批量操作

批量领取、指派、关闭等须 **部分成功**（TR-04）：

- 逐条校验版本或锁定状态；
- 单条冲突**不回滚**整批；
- 前端展示成功/失败摘要（**中文**）。
