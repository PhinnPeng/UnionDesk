## 二、背景与目标

### 2.1 背景

- 企业多品牌/多业务线并行运营，客户服务入口分散，工单与咨询数据无法统一治理。
- 各业务线对权限隔离、SLA、表单配置诉求不同，缺乏可复用的中台能力。

### 2.2 产品定位

UnionDesk 是一个企业级、多业务域、可配置的客户服务中台。它帮助企业统一管理**工单、在线咨询、反馈建议**三类服务请求，通过「业务域」实现不同品牌/业务线的独立运营与权限隔离。

### 2.3 核心价值

- **多业务域完全隔离**：一套系统，管理多个独立品牌，互不干扰。
- **统一身份，两端分离**：一个账号主体，可同时是 A 业务的客户、B 业务的客服。
- **动态工单表单**：各业务域可自定义工单字段和状态流程。
- **完整服务闭环**：在线咨询 → 转工单 → SLA 保障 → 满意度评价。

### 2.4 目标

| 指标维度 | 指标说明 | 目标值 |
| :--- | :--- | :--- |
| 业务域覆盖 | MVP 验收可闭环运行的业务域数量 | 待确认（验收基准：≥3） |
| 核心闭环 | 客户提单 → 客服处理 → 客户查单全流程可用率 | 待确认 |
| SLA 达标 | 配置 SLA 后，计时与预警准确率 | 待确认 |
| 权限隔离 | 跨域数据越权访问事件数 | 待确认（期望：0） |
| 性能 | 工单列表常用查询 P95 响应时间 | < 1s（见 §六.2） |

### 2.5 用户角色速览

| 角色 | 核心职责 | 工作端 | 对应系统角色 |
| :--- | :--- | :--- | :--- |

| **客服** | 处理工单、接待咨询、使用快捷回复 | 员工端 | `agent` |
| **域管理员** | 配置业务域（工单类型/SLA/角色/人员） | 域管理后台 | `super_admin` / `domain_admin` |

## 三、故事介绍

### 3.1 场景描述

- 客户在某业务域提交工单，填写动态表单，获得工单号并跟踪处理进度。
- 客户进入在线咨询窗口排队，接入客服后实时聊天，会话可一键转为工单。
- 客户在工单关闭或咨询结束后，通过通知中心完成星级与文字评价。
- 客服在工单队列中按域、状态、优先级、SLA 筛选并处理分配给自己的工单。
- 客服从咨询队列手动接入客户，聊天中可撤回 2 分钟内消息，并将会话转为工单。
- 域管理员使用低代码表单设计器配置工单类型、SLA 规则与通知模板。
- 域管理员管理本域成员、客户与自定义角色权限组合。
- 平台管理员创建/管理业务域，设置可见与注册策略，治理全局员工与安全策略。

### 3.2 价值分析

- 多业务域隔离使一套中台支撑多品牌独立运营，降低重复建设成本。
- 统一身份 + 两端分离，减少客户与员工账号割裂带来的体验与运维成本。
- 动态工单表单与 SLA 配置，使各域可按自身流程交付服务而不改代码。
- 咨询 → 工单 → SLA → 评价的闭环，提升服务可追踪性与满意度数据沉淀。

### 3.3 用户核心路径

**客户端主路径**

选择业务域 → 提交工单 / 进入在线咨询 → 等待处理 / 排队接入 → 查看进度 / 继续沟通 → 确认解决 → 满意度评价

**员工端主路径**

登录员工端 → 查看工单队列 / 咨询队列 → 领取或接入 → 处理（回复/转派/变更状态）→ 关闭工单 / 结束咨询

**域管理后台主路径**

进入本域 → 配置工单类型与 SLA → 管理成员与客户 → 分配角色权限

**平台管理后台主路径**

创建/管理业务域 → 管理全局员工与平台角色 → 配置全局安全策略与告警

### 3.4 漏斗目标

| 漏斗 | 步骤 | 转化率指标 | 目标值 |
| :--- | :--- | :--- | :--- |
| 客户提单 | 进入提单页 → 提交成功 | 提单提交率 | 待确认 |
| 在线咨询 | 进入咨询 → 排队接入成功 | 接入转化率 | 待确认 |
| 工单处理 | 工单创建 → 首次客服响应 | 首响达标率 | 待确认（与 SLA 规则联动） |
| 工单处理 | 工单创建 → 关闭 | 解决关闭率 | 待确认 |
| 满意度 | 可评价入口曝光 → 完成评价 | 评价参与率 | 待确认 |

### 3.5 路径规划（中长期）

- 知识库全流程
- Elasticsearch 全文搜索
- VitePress 帮助中心
- 反馈/建议独立轻量入口（三态展示）

## 四、概要设计

### 4.1 架构约束（全局设计决策）

#### 4.1.1 统一身份模型

- **一个身份主体**：`identity_subject` 是系统中自然人的唯一标识。一个手机号对应一个主体。
- **两种账号**：一个主体下，可同时拥有 `staff_account`（员工账号）和 `customer_account`（客户账号）。
- **登录与切换**：客户端和员工端分别登录，底层共享认证会话。若同一主体已登录员工端，打开客户端时可免密切换。
- **客户入域**：客户账号需加入某业务域后才能访问该域服务。入域方式由业务域的注册策略控制。

#### 4.1.2 多业务域隔离

业务域是资源隔离的基本单元。每个业务域拥有独立的工单、咨询、人员、配置。

| 策略类型 | 选项 | 说明 |
| :--- | :--- | :--- |
| **可见策略** | `公开` / `仅域内客户可见` / `仅渠道可见` | 控制谁可以发现这个域 |
| **注册策略** | `开放注册` / `仅邀请注册` / `仅管理员添加` | 控制客户如何加入这个域 |

#### 4.1.3 权限体系

平台采用 **「平台角色 + 业务域角色」双层权限**。

- **平台级**：`platform_admin`（平台管理员）拥有创建业务域、管理全局员工的最高权限。
- **业务域级**：每个业务域内置三个角色。
    - `super_admin`：域最高权限，管理域内一切。
    - `domain_admin`：管理域内常规配置与人员。
    - `agent`：基础客服，处理工单、接待咨询。
    - *域管理员可基于「权限项」组合创建最多 20 个自定义角色。*
- **权限项**：精确到按钮的功能权限，格式为 `模块:操作`，如 `ticket:delete`。数据范围通过 `:self` / `:domain_all` 后缀控制。

#### 4.1.4 全局安全策略

- **认证**：JWT + Refresh Token 轮换，支持单账号多会话踢除。
- **安全屏蔽**：全平台统一敏感词库，命中即拦截并告警。
- **审计**：所有管理操作、状态变更、权限修改均写入 `audit_log`，不可删除。

## 五、详细设计

### 5.1 客户端（用户视角）

*客户端是用户寻求帮助的统一入口。*

#### 5.1.1 服务请求

- **提交工单（F1.1）**：用户选择工单类型（问题/反馈/建议），填写动态表单，提交后获得工单号。反馈/建议类型可在域内预置为简表单工单类型，由管理员在工单类型设计中启用/停用（见 Backlog E2/E3）。
- **在线咨询（F1.2）**：用户进入咨询窗口，先排队，接入客服后可实时聊天。可发送文字、图片、附件。
- **反馈/建议（F1.3）**：**MVP 以工单类型路径交付**；独立轻量入口（三态展示）为可选形态，纳入下期规划，不与工单类型开关拆成两套后台模块。

#### 5.1.2 服务跟踪

- **我的工单（F1.4）**：查看提交过的所有工单，查看客服公开回复，补充信息，确认解决后评价。
- **咨询历史（F1.4）**：回看历史聊天记录和关联工单。
- **满意度评价（F1.5）**：工单关闭或咨询结束后，用户可在通知中心找到评价入口，进行星级+文字评价。

### 5.2 员工端（客服视角）

*客服在此处理分配给自己的工作。*

#### 5.2.1 工单处理中心

- **工单队列（F2.1）**：按域、状态、优先级、SLA 状态筛选工单。支持批量领取、批量关闭。
- **工单详情（F2.1）**：左侧时间线（公开回复、内部备注、系统操作），右侧属性卡（状态、优先级、SLA 倒计时）。
- **处理动作（F2.1）**：公开回复、内部备注、转派他人、变更状态、请求客户补充信息。
- **SLA 感知（F2.2）**：列表页和详情页高亮显示 SLA 即将超时或已超时的工单。

#### 5.2.2 在线咨询工作台

- **接待（F2.3）**：客服手动从队列中接入等待的客户。
- **聊天（F2.3）**：实时收发消息，支持撤回 2 分钟内的消息。
- **转工单（F2.3）**：一键将当前会话转为工单，自动带入客户信息和会话摘要。

### 5.3 域管理后台（管理员视角）

*域管理员在此配置本业务域的规则与人员。*

#### 5.3.1 域配置

- **工单类型设计（F3.1）**：使用低代码表单设计器创建/编辑工单类型，配置其动态字段和状态流程。
- **SLA 规则（F3.2）**：配置本域默认 SLA 和按工单类型的特殊 SLA（首响/解决时限、违约动作）。
- **通知模板（F3.2）**：编辑各类事件（工单创建、SLA 预警等）的站内信和邮件模板。

#### 5.3.2 人员与权限

- **成员管理（F3.3）**：从平台员工库中添加/移除本域成员，并分配角色。
- **客户管理（F3.3）**：查看本域客户列表，手动激活/禁用，打标签。
- **角色管理（F3.3）**：创建自定义角色，勾选所需的权限项组合。

### 5.4 平台管理后台（系统视角）

*平台管理员在此进行跨域的全局治理。*

#### 5.4.1 业务域管理

- **创建/管理（F4.1）**：新建业务域（从模板或空白），设置其可见与注册策略，启用/禁用。
- **模板中心（F4.2）**：将某个域的完整配置提炼为模板，供快速创建新域。

#### 5.4.2 员工与权限管理

- **员工账号（F4.3）**：全局员工列表，手动创建或 CSV 批量导入。分配平台角色，或将其加入指定业务域并赋予域角色。
- **离职池（F4.3）**：员工账号删除后进入离职池，其名下的未关闭工单自动退回队列。

#### 5.4.3 系统设置与安全

- **全局安全策略（F4.4）**：密码强度、登录失败锁定、IP 白名单。
- **安全告警中心（F4.4）**：查看和处理所有派生的安全告警（如暴力破解、高频操作）。

### 6.2 非功能要求

- **性能**：工单列表常用查询 P95 < 1s。
- **安全**：全站 HTTPS，密码 Argon2 哈希，JWT 认证。
- **终端**：客户端/员工端需响应式适配手机、平板、桌面端。
- **浏览器**：支持 Chrome/Edge/Firefox/Safari 最近两个主版本。

### 6.3 核心验收标准

- **核心闭环**：至少 3 个业务域下，客户提单 → 客服处理 → 客户查单的流程可闭环。
- **权限隔离**：A 域客服不能看到 B 域工单；自定义角色只能访问被授权的菜单和按钮。
- **安全验证**：关键操作（删除域、全量导出）需密码二次验证，并写入审计日志。
- **SLA 验证**：可配置工作日历，SLA 计时准确，临近/超时预警正常触发。
- **环境交付**：提供一键 `docker-compose.yml`，可跑通全部 P0 验收用例。

### A. 核心数据模型（简表）

*仅列出最关键的业务实体；逻辑模型见 [`architecture/data-model.md`](../architecture/data-model.md)；Sprint 表增量见 [`architecture/database-increment-plan.md`](../architecture/database-increment-plan.md)。*

| 实体 | 核心字段 | 说明 |
| :--- | :--- | :--- |
| **identity_subject** | id, phone, status | 统一身份主体 |
| **staff_account** | id, subject_id, login_name, status | 员工账号 |
| **customer_account** | id, subject_id, display_name, status | 客户账号 |
| **business_domain** | id, code, name, visibility_policy, registration_enabled, invitation_enabled | 业务域入域配置（逻辑字段见 foundation-rules §2.2） |
| **ticket** | id, ticket_no, domain_id, customer_id, assignee_id, status, sla_* | 工单主表 |
| **consultation_session** | id, domain_id, customer_id, agent_id, status | 咨询会话 |
| **domain_role** | id, domain_id, code, name, preset | 域内角色 |
